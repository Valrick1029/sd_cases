1. Уточнение целей и требований к продукту

Функциональные требования:
*   Извлечение информации из реляционной базы данных через естественный язык.
*   Поддержка сложных запросов с фильтрацией, агрегацией и соединением таблиц (JOIN).
*   Возврат ответа в текстовой форме с указанием источника данных.
*   Возможность работы с несколькими схемами базы данных.

Нефункциональные требования:
*   Задержка ответа (P99): не более 2 секунд.
*   Доступность: 99.9%.
*   Актуальность данных: не более 5 минут задержки от момента записи в БД до доступности в RAG.
*   Безопасность: разграничение доступа к данным на уровне пользователя.
*   Точность ответов: >90% корректных запросов к БД.

2. Требования к масштабируемости

*   Объем данных: Поддержка баз данных от 1 ГБ до 1 ТБ+.
*   Количество таблиц: От 10 до 1000+ таблиц в схеме.
*   Конкурентность: Обработка 100+ одновременных запросов.
*   Рост схемы: Возможность добавления новых таблиц и колонок без переобучения моделей.
*   Горизонтальное масштабирование: Возможность шардирования векторного индекса и репликации сервисов инференса.

3. Первичный анализ данных и определение ключевых признаков

*   Типы данных: Структурированные (числа, даты, категории) и текстовые поля.
*   Схема БД: Таблицы, колонки, типы данных, первичные и внешние ключи, индексы.
*   Семантика данных: Описания таблиц и колонок (метаданные), примеры значений.
*   Сложность запросов: Простые SELECT vs многоуровневые JOIN с агрегациями.
*   Качество данных: Наличие NULL-значений, дубликатов, неконсистентных форматов.
*   Частота обновлений: Статические справочники vs высокоизменчивые транзакционные данные.

4. Инфраструктура

Общие компоненты:
*   API Gateway: Маршрутизация запросов, аутентификация, авторизация, rate limiting.
*   LLM Serving: (vLLM, TGI) для хостинга языковой модели.
*   Vector Database: (Qdrant, Milvus, pgvector) для хранения эмбеддингов схемы и данных.
*   SQL Database: Источник истины (PostgreSQL, MySQL, ClickHouse).
*   Cache Layer: (Redis) для кэширования частых запросов и промежуточных результатов.
*   Message Queue: (Kafka) для асинхронной синхронизации данных.

Специфика по вариантам:
*   Для Text-to-SQL: Требуется доступ к метаданным схемы в реальном времени.
*   Для embedding-based retrieval: Требуется пайплайн векторизации данных при изменении в БД.

5. Выбор и разработка моделей

Вариант 1: Text-to-SQL (Генерация запросов)

*   Принцип: LLM генерирует SQL-запрос на основе вопроса пользователя и схемы БД.
*   Архитектура: LLM + Schema Retriever + SQL Executor + Result Formatter.
*   Добавление таблиц: Автоматическое обновление метаданных схемы в контексте модели.
*   Преимущества: Точные ответы на основе актуальных данных, поддержка сложных агрегаций.
*   Недостатки: Риск генерации некорректного SQL, требуется валидация запросов.
*   Задержка: 500-1500 мс.

Вариант 2: Embedding-Based Retrieval (Векторный поиск по данным)

*   Принцип: Данные из таблиц векторизуются и хранятся в векторной БД. Поиск по эмбеддингу вопроса.
*   Архитектура: Embedding Model + Vector DB + LLM для синтеза ответа.
*   Добавление таблиц: Требуется переиндексация данных новой таблицы.
*   Преимущества: Быстрый поиск, не зависит от сложности схемы БД.
*   Недостатки: Данные могут устаревать, не подходит для агрегаций и вычислений.
*   Задержка: 200-500 мс.

Вариант 3: Гибридный подход (Text-to-SQL + Vector Retrieval)

*   Принцип: Маршрутизатор определяет тип запроса. Для фактологических — векторный поиск, для аналитических — Text-to-SQL.
*   Архитектура: Router + (Text-to-SQL Pipeline | Vector Retrieval Pipeline) + Response Synthesizer.
*   Добавление таблиц: Обновление метаданных для Text-to-SQL, переиндексация для Vector.
*   Преимущества: Универсальность, оптимальный выбор инструмента под задачу.
*   Недостатки: Сложность архитектуры, требуется обучение маршрутизатора.
*   Задержка: 300-1500 мс (зависит от выбранного пути).

Вариант 4: Semantic Layer + SQL (Предварительно определённые метрики)

*   Принцип: Семантический слой мапит вопросы пользователя на предопределённые метрики и измерения.
*   Архитектура: Semantic Layer (dbt, Cube.js) + LLM для маппинга + SQL Generator.
*   Добавление таблиц: Требуется ручное или полуавтоматическое определение метрик.
*   Преимущества: Высокая точность, контроль над бизнес-логикой, безопасность.
*   Недостатки: Требует ручной настройки, менее гибкий для ad-hoc запросов.
*   Задержка: 300-800 мс.

Вариант 5: Graph-Based RAG (Графовая структура данных)

*   Принцип: Схема БД представляется как граф (таблицы — узлы, связи — рёбра). Поиск пути между сущностями.
*   Архитектура: Graph DB (Neo4j) + LLM для навигации по графу + Query Generator.
*   Добавление таблиц: Добавление узлов и рёбер в граф схемы.
*   Преимущества: Эффективен для сложных связей между таблицами, визуализация путей.
*   Недостатки: Сложность построения графа, требует дополнительных инструментов.
*   Задержка: 400-1000 мс.

Вариант 6: Multi-Agent Architecture (Агентный подход)

*   Принцип: Несколько специализированных агентов (Schema Explorer, Query Writer, Validator, Answerer) работают совместно.
*   Архитектура: Orchestrator + Specialist Agents + Shared Memory.
*   Добавление таблиц: Агенты динамически обнаруживают изменения схемы.
*   Преимущества: Высокая надёжность за счёт валидации, возможность исправления ошибок.
*   Недостатки: Высокая задержка из-за множественных вызовов LLM, сложность отладки.
*   Задержка: 1000-3000 мс.

Рекомендация: Для большинства сценариев оптимальным является Вариант 3 (Гибридный подход). Вариант 1 подходит для аналитических задач с чёткой схемой. Вариант 2 — для быстрого поиска по текстовым данным. Вариант 6 применим для критически важных систем где точность приоритетнее задержки.

6. Контроль инференса

*   Валидация SQL-запросов: Синтаксическая проверка перед исполнением, ограничение на опасные операции (DROP, DELETE, UPDATE).
*   Мониторинг точности: Отслеживание процента успешных запросов к БД, анализ ошибок генерации SQL.
*   A/B тестирование: Сравнение разных моделей и подходов на исторических запросах.
*   Human-in-the-loop: Запросы с низкой уверенностью или ошибками исполнения направляются на ручную проверку.
*   Логирование: Сохранение вопросов, сгенерированных SQL-запросов, результатов и ошибок для отладки.
*   Алертинг: Уведомления при росте процента ошибок SQL, увеличении задержки, падении доступности БД.
*   Кэширование запросов: Кэширование частых вопросов для снижения нагрузки на LLM и БД.
*   Audit Log: Полное логирование доступа к данным для compliance и безопасности.
*   Дрейф схемы: Мониторинг изменений в структуре БД и автоматическое обновление метаданных.
