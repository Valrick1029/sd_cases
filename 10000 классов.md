1. Уточнение целей и требований к продукту

Функциональные требования:
*   Классификация текстовых запросов по одному из 10 000 существующих классов.
*   Возможность добавления новых классов без остановки сервиса и полного переобучения модели.
*   Возврат наиболее вероятного класса и оценки уверенности (confidence score).

Нефункциональные требования:
*   Задержка инференса (P99): не более 100 мс.
*   Доступность: 99.9%.
*   Время интеграции нового класса: менее 1 минуты (обновление метаданных).
*   Точность (Accuracy): сопоставима со статическим классификатором (целевой показатель >85%).

2. Требования к масштабируемости

*   Рост количества классов: Архитектура должна поддерживать масштабирование от 10 000 до 100 000+ классов без линейного роста задержки инференса.
*   Нагрузка (QPS): Ожидаемая пиковая нагрузка должна быть распределена между узлами инференса. Горизонтальное масштабирование сервисов обработки запросов.
*   Объем данных: Хранилище должно вмещать векторные представления всех классов и историю запросов для дообучения.
*   Обновление модели: Процесс обновления весов модели или индексации векторов не должен блокировать поток запросов (Blue-Green deployment или горячая перезагрузка индексов).

3. Первичный анализ данных и определение ключевых признаков

*   Входные данные: Текстовые строки переменной длины. Требуется нормализация (lowercasing, удаление стоп-слов, лемматизация) в зависимости от выбранной модели.
*   Представление классов: Каждый класс должен иметь текстовое описание или набор примеров (shots) для формирования эталонного вектора.
*   Ключевые признаки: Семантическая близость текста запроса к описанию класса. Синтаксические признаки вторичны.
*   Проблема дисбаланса: Новые классы могут не иметь обучающей выборки (zero-shot) или иметь мало примеров (few-shot).

4. Инфраструктура

Общие компоненты:
*   API Gateway: Маршрутизация запросов, rate limiting, аутентификация.
*   Feature Store: Хранение предвычисленных эмбеддингов для классов.
*   Vector Database: (Milvus, Qdrant, Elasticsearch) для хранения векторов классов и поиска ближайших соседей (ANN).
*   Model Serving: (TorchServe, Triton) для хостинга моделей кодирования текста.
*   Message Queue: (Kafka, RabbitMQ) для асинхронной обработки задач дообучения и логирования.

5. Выбор и разработка моделей

Вариант 1: Метрическое обучение (Semantic Search / Prototype Network)

*   Принцип: Модель кодирует текст запроса и текстовое описание класса в единое векторное пространство. Классификация сводится к поиску ближайшего вектора класса (Cosine Similarity).
*   Архитектура: Siamese Network или двуэнкодерная модель (Bi-Encoder, например, SBERT).
*   Добавление класса: Вычисляется эмбеддинг описания нового класса и сохраняется в векторную базу. Переобучение модели не требуется.
*   Преимущества: Мгновенное добавление классов, линейная масштабируемость поиска через ANN.
*   Недостатки: Может уступать в точности задачам с тонкими семантическими различиями по сравнению с кросс-энкодерами.
*   Задержка инференса: 10-50 мс (зависит от размера индекса).

Вариант 2: Инкрементальное обучение (Incremental Fine-tuning)

*   Принцип: Классический классификатор (например, на базе Transformer) с динамическим выходным слоем.
*   Архитектура: Pre-trained Language Model + Classification Head.
*   Добавление класса: Требуется дообучение модели на примерах нового класса (Few-shot fine-tuning) с использованием методов предотвращения катастрофического забывания (EWC, Replay Buffer).
*   Преимущества: Высокая точность за счет учета контекста взаимодействия классов.
*   Недостатки: Риск деградации качества на старых классах, задержка на внедрение нового класса (время обучения), сложность инфраструктуры обновления.
*   Задержка инференса: 20-80 мс.

Вариант 3: Иерархическая классификация (Hierarchical Classification)

*   Принцип: Классы организованы в древовидную структуру. Классификация происходит поэтапно: от корня к листу.
*   Архитектура: Каскад моделей, где каждая модель отвечает за свой уровень иерархии.
*   Добавление класса: Новый класс добавляется в соответствующую ветку дерева. Требуется переобучение только модели данного уровня.
*   Преимущества: Снижение сложности задачи для каждой модели, возможность использования разных моделей для разных уровней, эффективный поиск (O(log N)).
*   Недостатки: Требуется заранее определенная иерархия, ошибка на верхнем уровне необратима, сложность поддержки актуальности дерева.
*   Задержка инференса: 30-70 мс (сумма задержек всех уровней).

Вариант 4: Двухэтапная классификация (Coarse-to-Fine / Retrieve-then-Rerank)

*   Принцип: Первый этап быстро отбирает N кандидатов (например, 100), второй этап точно ранжирует кандидатов.
*   Архитектура: Bi-Encoder (быстрый поиск) + Cross-Encoder (точное ранжирование).
*   Добавление класса: Новый класс добавляется в индекс первого этапа. Второй этап не требует переобучения.
*   Преимущества: Баланс между скоростью и точностью, Cross-Encoder дает более качественные оценки схожести.
*   Недостатки: Две модели в пайплайне увеличивают общую задержку, сложнее отладка.
*   Задержка инференса: 50-100 мс (10-30 мс检索 + 40-70 мс ранжирование).

Вариант 5: Классификация на основе LLM (Prompt-Based / In-Context Learning)

*   Принцип: Использование большой языковой модели с контекстным обучением. Классы передаются в промпте.
*   Архитектура: LLM API (GPT, Claude, LLaMA) + динамическое формирование промпта с описанием классов.
*   Добавление класса: Новый класс добавляется в список доступных классов в промпте.
*   Преимущества: Zero-shot классификация без обучения, высокая гибкость, понимание сложных нюансов.
*   Недостатки: Высокая стоимость запроса, высокая задержка (500-2000 мс), ограничение на длину контекста (не все 10 000 классов поместятся).
*   Задержка инференса: 500-2000 мс.
*   Применение: Подходит для офлайн-обработки или как второй этап в двухэтапной архитектуре.

Вариант 6: Гибридный подход (Ensemble / Multi-Model Voting)

*   Принцип: Комбинация нескольких моделей разных типов для повышения устойчивости предсказания.
*   Архитектура: Параллельный инференс 2-3 моделей + агрегатор решений (voting, weighted average, stacking).
*   Добавление класса: Зависит от базовых моделей (для метрических — мгновенно, для обучаемых — с дообучением).
*   Преимущества: Повышенная точность и устойчивость к ошибкам отдельных моделей, возможность постепенной миграции на новые модели.
*   Недостатки: Увеличение вычислительных затрат в 2-3 раза, сложность согласования выходов моделей.
*   Задержка инференса: 50-150 мс (определяется самой медленной моделью).

Рекомендация: Для возможности добавления нового класса в любой момент с балансом точности и задержки оптимальным является Вариант 4 (Двухэтапная классификация). Вариант 1 подходит для бюджетных решений. Вариант 5 применим как резервный механизм для сложных случаев.

6. Контроль инференса

*   Мониторинг качества: Отслеживание распределения уверенности (confidence distribution). Рост количества запросов с низкой уверенностью сигнализирует о появлении новых неучтенных тем или дрейфе данных.
*   A/B тестирование: Параллельный прогон трафика на старой и новой версии модели/индекса перед полным переключением.
*   Human-in-the-loop: Запросы с низкой уверенностью направляются на ручную разметку. Размеченные данные используются для валидации новых классов или дообучения.
*   Логирование: Сохранение входных запросов, предсказаний и векторов для последующего анализа ошибок и отладки.
*   Алертинг: Уведомления при росте latency выше порога или падении доступности векторной базы.
*   Дрейф данных: Регулярная проверка распределения входных запросов на предмет концептуального дрейфа (data drift detection).
*   Канареечные развертывания: Постепенное направление трафика на новую версию модели (1% → 5% → 25% → 100%) с автоматическим откатом при деградации метрик.
