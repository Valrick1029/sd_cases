1. Уточнение целей и требований к продукту

Функциональные требования:
*   Автоматическое извлечение персональных данных из диалога пользователя с языковой моделью.
*   Структурированное хранение извлечённых данных (профиль пользователя).
*   Обновление профиля по мере поступления новой информации из диалогов.
*   Предоставление доступа к профилю для персонализации ответов модели.
*   Управление согласием пользователя на сбор и хранение данных.
*   Возможность просмотра, экспорта и удаления пользователем своих данных.

Нефункциональные требования:
*   Задержка записи данных: асинхронно, без влияния на latency диалога.
*   Задержка чтения профиля: не более 100 мс перед генерацией ответа.
*   Доступность: 99.9%.
*   Соответствие регуляторным требованиям: GDPR, CCPA, 152-ФЗ.
*   Безопасность: шифрование данных на rest и in transit, аудит доступа.
*   Прозрачность: пользователь должен знать, какие данные собираются.

Правовые ограничения:
*   Явное согласие пользователя на сбор персональных данных из диалога.
*   Право на удаление (right to be forgotten).
*   Право на доступ и исправление данных.
*   Запрет на сбор чувствительных данных без явного отдельного согласия.
*   Возможность отключения сбора данных в любой момент.

2. Требования к масштабируемости

*   Количество пользователей: От 100 000 до 10 млн+ профилей.
*   Объём данных на профиль: От 1 КБ до 1 МБ (структурированные данные + история извлечений).
*   Частота обновлений: До 100 обновлений профиля на пользователя в месяц.
*   Конкурентность: 10 000+ одновременных диалогов.
*   Объём диалогов: До 1 млн сообщений в час для обработки.
*   Географическое распределение: Данные должны храниться в регионе проживания пользователя (data residency).
*   Горизонтальное масштабирование: Шардирование по user_id, репликация для чтения.

3. Первичный анализ данных и определение ключевых признаков

Категории извлекаемых данных:

| Категория        | Примеры                           | Метод извлечения               | Чувствительность |
| ---------------- | --------------------------------- | ------------------------------ | ---------------- |
| Идентификаторы   | Имя, фамилия, псевдоним           | NER (Named Entity Recognition) | Высокая          |
| Контакты         | Email, телефон, адрес             | Regex + NER                    | Высокая          |
| Демография       | Возраст, пол, город, страна       | NER + Classification           | Средняя          |
| Профессиональные | Должность, компания, индустрия    | NER + Relation Extraction      | Средняя          |
| Образование      | ВУЗ, степень, год выпуска         | NER + Relation Extraction      | Низкая           |
| Предпочтения     | Любимые продукты, хобби, интересы | Intent Classification          | Низкая           |
| Поведенческие    | Стиль общения, тон, частота       | Behavioral Analytics           | Низкая           |
| Контекстуальные  | Текущие задачи, проекты, цели     | Summarization + Extraction     | Средняя          |

Ключевые признаки:
*   Уверенность извлечения: Confidence score для каждого извлечённого факта (0.0–1.0).
*   Источник: ID диалога, timestamp, сообщение.
*   Актуальность: Временная метка последнего подтверждения факта.
*   Статус верификации: Unverified / User-confirmed / Auto-verified.
*   Согласие: Статус согласия на хранение каждого типа данных.
*   История изменений: Audit trail всех обновлений профиля.

Проблемы качества:
*   Ложные извлечения (модель поняла контекст неверно).
*   Устаревшая информация (пользователь сменил работу).
*   Конфликты данных (противоречивые утверждения в разных диалогах).
*   Гипотетические утверждения (пользователь говорит «если бы я был...»).
*   Контекстуальная зависимость (шутки, ролевые сценарии).

4. Инфраструктура

Общие компоненты:
*   Chat Interface: Веб/мобильный клиент для диалога с моделью.
*   LLM Gateway: Проксирование запросов к LLM, инъекция контекста профиля.
*   Data Extraction Pipeline:
    *   Stream Processor (Kafka Streams / Flink) — обработка сообщений в реальном времени.
    *   NER Service — извлечение сущностей.
    *   Confidence Scorer — оценка достоверности.
*   Profile Storage:
    *   PostgreSQL — структурированный профиль пользователя.
    *   MongoDB — гибкие атрибуты и история извлечений.
    *   Redis — кэш горячих профилей.
*   Consent Management Service: Управление согласиями, отзывами, настройками приватности.
*   Encryption Service: Прозрачное шифрование чувствительных полей (KMS).
*   Audit Service: Логирование всех операций с данными.

Специфика по вариантам:
*   Для real-time персонализации: Требуется синхронное чтение профиля перед генерацией ответа.
*   Для асинхронного сбора: Обработка диалогов постфактум без влияния на latency.

5. Выбор и разработка моделей

Вариант 1: Rule-Based Extraction (Правила + Regex + NER)

*   Принцип: Предопределённые правила и модели NER для извлечения сущностей.
*   Архитектура: Message -- NER Model -- Rule Engine -- Profile Update.
*   Добавление категорий: Требуется обновление правил и переобучение NER.
*   Преимущества: Предсказуемость, контроль, низкая стоимость.
*   Недостатки: Низкая гибкость, пропуск неявных упоминаний.
*   Задержка: 10-50 мс на сообщение.

Вариант 2: LLM-Based Extraction (Генерация структурированных данных)

*   Принцип: LLM анализирует сообщение и извлекает данные в структурированном формате (JSON).
*   Архитектура: Message -- LLM (Extraction Prompt) -- JSON Validator -- Profile Update.
*   Добавление категорий: Обновление промпта, без переобучения.
*   Преимущества: Высокая гибкость, понимание контекста, извлечение неявных фактов.
*   Недостатки: Высокая стоимость, риск галлюцинаций, задержка.
*   Задержка: 200-500 мс на сообщение.

Вариант 3: Гибридный подход (LLM + Validation Layer)

*   Принцип: LLM извлекает данные, валидационный слой проверяет достоверность.
*   Архитектура: Message -- LLM Extractor -- Confidence Scorer -- Rule Validator -- Profile.
*   Добавление категорий: Обновление промпта и правил валидации.
*   Преимущества: Баланс гибкости и надёжности, снижение ложных срабатываний.
*   Недостатки: Сложнее архитектура, две модели в пайплайне.
*   Задержка: 250-600 мс на сообщение.

Вариант 4: Асинхронная обработка (Batch Processing)

*   Принцип: Диалоги накапливаются, обработка происходит периодически (раз в час/день).
*   Архитектура: Message Queue -- Batch Job -- Extraction Model -- Profile Update.
*   Добавление категорий: Обновление batch-пайплайна.
*   Преимущества: Низкая стоимость, возможность использования тяжёлых моделей.
*   Недостатки: Данные не актуальны для текущей сессии, задержка обновления.
*   Задержка: Не влияет на диалог (асинхронно).

Вариант 5: User-Confirmed Extraction (Верификация пользователем)

*   Принцип: Модель явно запрашивает подтверждение перед сохранением факта.
*   Архитектура: Message -- LLM Extractor -- Confirmation Dialog -- Profile Update.
*   Добавление категорий: Обновление промпта для confirmation.
*   Преимущества: Максимальная достоверность, прозрачность для пользователя.
*   Недостатки: Ухудшает UX (лишние вопросы), прерывает поток диалога.
*   Задержка: 500-1000 мс + время на ответ пользователя.

Вариант 6: Контекстуальная память (Session-Based + Long-Term Memory)

*   Принцип: Разделение на краткосрочную (сессия) и долгосрочную (профиль) память.
*   Архитектура: Session Store (Redis) + Long-Term Profile (PostgreSQL) + Memory Manager.
*   Добавление категорий: Обновление схемы памяти.
*   Преимущества: Гибкое управление контекстом, разделение временных и постоянных данных.
*   Недостатки: Сложность управления жизненным циклом данных.
*   Задержка: 50-200 мс.

Сравнительная таблица:

| Вариант | Точность | Задержка | Стоимость | UX |
|---------|----------|----------|-----------|-----|
| Rule-Based | Средняя | Низкая | Низкая | Прозрачный |
| LLM-Based | Высокая | Средняя | Высокая | Прозрачный |
| Гибридный | Очень высокая | Средняя | Средняя | Прозрачный |
| Асинхронный | Высокая | Нет | Низкая | Невидимый |
| User-Confirmed | Очень высокая | Высокая | Средняя | Навязчивый |
| Контекстуальный | Высокая | Низкая | Средняя | Прозрачный |

Рекомендация: Для баланса точности, стоимости и UX оптимален Вариант 3 (Гибридный) с асинхронной обработкой (Вариант 4) для не критичных данных. Вариант 5 применим для чувствительных данных. Вариант 6 рекомендуется как дополнение к любому из вариантов для управления контекстом сессии.

6. Контроль инференса

*   Согласие и compliance: Автоматическая проверка статуса согласия перед каждым извлечением и сохранением. Журнал отзывов согласия.
*   Аудит доступа: Полное логирование кто (какой сервис), когда, к каким данным обращался.
*   Мониторинг качества: Отслеживание confidence score извлечений, процента отклонённых пользователем фактов.
*   Data Quality Pipeline: Автоматическое выявление конфликтов данных, устаревших записей, аномалий.
*   Право на удаление: Автоматизированный пайплайн удаления всех данных пользователя по запросу (включая бэкапы и логи).
*   Прозрачность для пользователя: Интерфейс просмотра всех сохранённых данных с возможностью редактирования и удаления.
*   Шифрование: Прозрачное шифрование чувствительных полей (имя, контакты), хранение ключей в KMS.
*   Алертинг: Уведомления о несанкционированном доступе, аномальной активности, массовых отказах пользователей.
*   Retention Policy: Автоматическое удаление неактивных профилей по истечении срока (например, 2 года без активности).
*   Data Residency: Контроль географического расположения данных и маршрутизация запросов в соответствующий регион.
*   Model Drift: Регулярная оценка качества извлечения на размеченном датасете, переобучение при деградации.
